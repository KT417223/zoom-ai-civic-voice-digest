# Project Context

このドキュメントは、別マシンや別スレッドのCodexで作業を再開するときのための引き継ぎメモです。

## 目的

Qiita公式イベント「Zoom AI Servicesで文字起こし、翻訳、要約をやってみよう！」への応募を見据えた検証プロジェクトです。

狙いは、自治体が公開している「声の広報」「市報音声版」を題材に、Zoom AI Services を使った多言語ダイジェスト化の設計を検証することです。既存音声を直接扱えるとは限らないため、当面は文字起こし済みテキストを主入力にします。

## 記事の方向性

第一候補のタイトル:

```text
自治体の音声広報を多言語ダイジェスト化する設計をZoom AI Servicesで考えてみた
```

問題設定:

```text
自治体の音声版広報はアクセシビリティに役立つ一方で、あとから検索したり、重要なお知らせだけ拾ったり、多言語で把握したりするには手間がある。
既存音声を直接Zoom AI Servicesに渡せるとは限らない前提で、文字起こし済みテキストを入力にして、要約、翻訳、構造化出力をつなぐ設計を検証する。
```

## 選出基準との対応

Qiitaイベントの選出基準に合わせて、記事では以下を意識する。

- アイデアの独自性: 自治体の音声広報を多言語ダイジェスト化する
- 実用性: 住民向け情報アクセス、アクセシビリティ、多言語対応に効く
- 再現性: Web上で公開されている音声URLを題材にする
- 検証視点: 固有名詞、日付、期限、対象者、要約品質、コストを見る

## 権利・引用まわりの方針

「引用した」と強く言うより、以下の表現を使う。

```text
公開されている音声版広報を、技術検証の題材として参照した。
```

記事に入れる予定の注意書き:

```md
本記事では、自治体がWeb上で公開している音声版広報を技術検証の題材として参照します。
音声ファイル自体の再配布は行わず、元ページへのリンク、処理手順、APIの出力例を中心に紹介します。
各コンテンツの利用条件は、各自治体サイトの利用規約に従ってください。
```

避けること:

- 音声ファイルをリポジトリにコミットしない
- 広報本文や全文文字起こしを丸ごと転載しない
- APIキーや取得済みの音声ファイルをコミットしない
- 自治体公式の見解のように見える書き方をしない

## 想定する処理フロー

```text
既存文字起こしテキスト
→ 重要なお知らせ、期限、対象者、手続きを抽出
→ 英訳
→ Markdown / JSON で出力
```

将来的にZoom AI Services側で任意の既存音声を扱えることが確認できた場合のみ、公開音声URLや手元ファイルを入力アダプタとして追加する。

## 仕様確認で注意すること

2026年7月時点で要確認の大きなリスク:

- Zoom AI Services / Scribe が、任意の既存MP3 URLや手元音声ファイルを直接アップロードして文字起こしできるとは限らない
- Zoom会議、Zoom録画、またはZoom内で生成された文字起こし・要約を主対象にしている可能性がある
- 公開自治体音声をそのままScribeに投入できない場合、記事の主軸を変える必要がある

代替案:

1. Zoomで開催された自治体説明会・庁内会議の録画/文字起こしを多言語ダイジェスト化する想定に寄せる
2. 既存音声の文字起こしは別手段で行い、Zoom AI Servicesでは要約・翻訳部分を検証する
3. 「既存公開音声をZoom AI Servicesだけで処理しようとして分かった制約」を記事の検証ポイントにする

入力方法の方針:

- MVPは `--transcript-file` を優先する
- `--audio-url` は将来の入力候補として残す
- 既に文字起こしがある場合は、音声処理をスキップして要約・翻訳だけ実行できるようにする
- 実際の音声対応形式はZoom AI ServicesのAPI仕様を確認してから確定する

最終的な出力イメージ:

- 日本語ダイジェスト
- 英語ダイジェスト
- JSON形式の構造化データ
- Qiita記事用の短い出力例

## 評価観点

- 固有名詞: 地名、施設名、制度名を正しく拾えるか
- 日付・期限: 申込期限、開催日を落とさないか
- 対象者: 高齢者、子育て世帯、市内在住者などを抽出できるか
- 要約品質: 長い音声から必要な情報だけ読めるか
- 翻訳品質: 自治体情報として自然な英語になっているか
- コスト: 1ファイルあたりの処理時間と概算費用
- 再利用性: 他自治体にも同じ処理を使えるか
